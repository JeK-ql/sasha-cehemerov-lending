# Чекаут-форма — кількість, доставка Нової Пошти, екран подяки

**Дата:** 2026-05-22
**Проєкт:** Лендинг Sasha Chemerov (`isusneisus.com`)
**Тип:** Доопрацювання наявної чекаут-модалки

---

## 1. Мета

Розширити форму замовлення в чекаут-модалці чотирма змінами:

1. Степер `+ / −` — замовлення кількох одиниць товару.
2. Зручніший вибір міста Нової Пошти — список популярних міст.
3. Вибір точки доставки — відділення / поштомат / кур'єр на адресу.
4. Екран «Дякую за замовлення» після натискання кнопки замовлення.

Плюс наскрізне покращення форми: плейсхолдери, підказки під полями та
валідація через Zod з українськими повідомленнями про помилки.

---

## 2. Обсяг

**Входить:** усе з розділів 3–8 нижче.

**Не входить:**

- Зміна Telegram-нотифікації менеджерам. Вона спрацьовує на серверному
  колбеку WayForPay і лишається як є.
- Передача полів доставки (місто / відділення / адреса кур'єра) кудись далі
  за валідацію. Сьогодні ці поля збираються й валідуються, але не доходять
  до жодного отримувача (Telegram читає колбек WayForPay, де їх немає).
  Така поведінка зберігається — це окреме питання поза цим скоупом.
- Адресна доставка інших служб (Укрпошта тощо).

## 3. Кількість товару (степер + / −)

### Поведінка

- У блоці зведення замовлення (`.order`) замість статичного `×1` —
  степер: кнопка `−`, поточне число, кнопка `+`.
- Мінімум — **1**. Верхньої межі немає. Кнопка `−` неактивна на значенні 1;
  кнопка `+` активна завжди.
- Рядок ціни показує суму за всю кількість: `quantity × 2200 ₴`
  (напр. `2 × 2200 = 4400 ₴`).

### Технічно

- `CheckoutInput` отримує поле `quantity: number`.
- `/api/checkout`: `amount = PRODUCT.price × quantity`,
  `productPrice = [PRODUCT.price]`, `productCount = [quantity]`.
  Функція `purchaseSignature` уже коректно розгортає масиви товарів —
  логіку підпису міняти не треба.
- Новий компонент `QuantityStepper` (props: `value: number`,
  `onChange: (n: number) => void`).

## 4. Форма: плейсхолдери, підказки, Zod-валідація

### Плейсхолдери

Кожен інпут отримує плейсхолдер-приклад:

| Поле     | Плейсхолдер           |
| -------- | --------------------- |
| Прізвище та ім'я | `Чемеров Олександр` |
| Телефон  | `+380 67 123 45 67`   |
| E-mail   | `you@example.com`     |
| Місто    | `Почніть вводити назву` |
| Вулиця   | `вул. Шевченка`       |
| Будинок  | `12А`                 |
| Квартира | `45`                  |

### Підказки під полями

Під кожним полем — рядок-підказка (як зараз під e-mail), що пояснює,
що саме вписати. Стиль — наявний клас `.fieldHint`.

### Валідація через Zod

- Додати `zod` у залежності (`package.json`).
- Новий файл `lib/checkoutSchema.ts` — Zod-схема `checkoutSchema` плюс
  виведений тип. Українські повідомлення про помилки.
- Схема — єдине джерело правди: її використовує і клієнтська форма,
  і серверний `/api/checkout`.
- Файл `lib/validate.ts` (саморобна `validateCheckout`) видаляється;
  маршрут `/api/checkout` переходить на `checkoutSchema.safeParse`.

**Правила й повідомлення:**

| Поле        | Правило                                  | Повідомлення укр.                       |
| ----------- | ---------------------------------------- | --------------------------------------- |
| `fullName`  | щонайменше два слова                     | `Вкажіть прізвище та ім'я`              |
| `phone`     | `^\+?\d{9,15}$` (без пробілів)           | `Невірний номер телефону`               |
| `email`     | формат e-mail                            | `Невірний e-mail`                       |
| `quantity`  | ціле число ≥ 1                           | `Кількість має бути не менше 1`         |
| `city`      | непорожнє                                | `Оберіть місто`                         |
| `warehouse` | непорожнє, **якщо** `deliveryType=warehouse` | `Оберіть відділення або поштомат`   |
| `street`    | непорожнє, **якщо** `deliveryType=courier`   | `Вкажіть вулицю`                    |
| `building`  | непорожнє, **якщо** `deliveryType=courier`   | `Вкажіть будинок`                   |
| `flat`      | необов'язкове                            | —                                       |

Умовні поля доставки реалізуються через `superRefine` у схемі.

### Показ помилок на клієнті

- До першої спроби «Замовити» помилки не показуються.
- Після натискання «Замовити» з невалідними даними: під кожним
  проблемним полем замість підказки з'являється **червоний текст
  помилки**. Форма не відправляється.
- Після першої спроби кожне поле перевалідовується на зміну —
  помилка зникає, щойно поле стає валідним.
- Кнопка «Замовити» лишається доступною завжди (валідація — на кліку),
  щоб користувач бачив, які поля заповнити.

## 5. Популярні міста Нової Пошти

### Поведінка

- Новий файл `lib/popularCities.ts` — константа `POPULAR_CITIES`:
  масив `{ label, ref }` для 8 міст — **Київ, Харків, Одеса, Львів,
  Дніпро, Запоріжжя, Вінниця, Полтава**.
- `ref` — справжні settlement-Ref Нової Пошти. Отримуються одноразово
  через API Нової Пошти під час реалізації та зашиваються у константу.
- **Без кешування і без окремого API-ендпоінта.** Просто константа.
- Поле «Місто» у фокусі й порожнє → випадає список `POPULAR_CITIES`
  із заголовком «Популярні міста». Клік по місту обирає його одразу
  (label + ref), без додаткового запиту.
- Введено **2+ символи** → список перемикається на звичайний живий
  пошук через наявний `/api/novaposhta?type=cities` (поточна логіка).
- Очищення поля → знову показуються популярні міста.

Список рендериться наявним стилем `.ac` / `.acItem`.

## 6. Точка доставки

Після того як обрано місто, з'являється перемикач режиму доставки —
сегментний контрол на два режими.

### Режим «Відділення / поштомат» (за замовчуванням)

- Поле з пошуком + випадний фільтрований список точок видачі.
- Точки тягнуться через наявний `/api/novaposhta?type=warehouses&ref=`.
- **Сортування: відділення першими, потім поштомати**; усередині групи —
  за номером.
- Кожен рядок списку має **мітку типу** — «Відділення» або «Поштомат».
- `lib/novaposhta.ts`: тип `NpWarehouse { label, ref, type, number }`,
  де `type: 'branch' | 'postbox'`. `mapWarehouses` визначає `type` з поля
  `CategoryOfWarehouse` Нової Пошти (фолбек — наявність слова «Поштомат»
  у назві) і повертає відсортований масив.

### Режим «Кур'єр»

- Три текстові поля: **Вулиця**, **Будинок**, **Квартира**
  (квартира — необов'язкова).
- Кур'єрська доставка Нової Пошти на адресу.

### Скидання стану

- Зміна міста → скидаються і вибране відділення, і поля кур'єра.
- Перемикання режиму доставки → поля неактивного режиму очищуються.

## 7. Оплата і екран подяки

### Потік

Оплата — **лише через WayForPay**.

1. Користувач натискає кнопку замовлення.
2. Запускається Zod-валідація. Невалідно → лишаємось на формі,
   показуємо помилки (розділ 4).
3. Валідно → `POST /api/checkout`, отримуємо параметри WayForPay.
4. Одночасно:
   - вміст модалки замінюється на **екран подяки**;
   - запускається віджет оплати WayForPay (`new window.Wayforpay().run(params)`)
     поверх модалки.
5. Користувач оплачує у віджеті WayForPay. Коли віджет закривається —
   під ним лишається екран подяки.

Екран подяки показується одразу на кліку — окремо слухати подію
`approved` віджета не потрібно. Джерело правди про оплату — серверний
колбек WayForPay (`/api/wayforpay-callback`), він і далі тригерить
Telegram-нотифікацію.

### Екран подяки — `OrderSuccess`

- Новий компонент `OrderSuccess` (props: `orderReference: string`,
  `onClose: () => void`).
- Вміст: великий заголовок **«ДЯКУЮ ЗА ЗАМОВЛЕННЯ»**, підзаголовок
  **«Очікуйте на товар»**, номер замовлення, кнопка **«Закрити»**.
- Стиль — як на сайті: брутал-естетика темної модалки, наявні
  типографічні класи й токени.
- Рендериться всередині модалки замість форми.

### Скидання

- `CheckoutForm` тримає стан «замовлення оформлено» (`submitted` + `orderRef`)
  і рендерить `OrderSuccess` замість форми.
- Закриття модалки розмонтовує `CheckoutForm` (наявна логіка
  `CheckoutModal` із `EXIT_MS`), тож наступне відкриття — чиста форма.

### Кнопка

- Текст кнопки — **«ЗАМОВИТИ»**, підпис під ним —
  `ОПЛАТА КАРТКОЮ · WAYFORPAY · APPLE PAY · GOOGLE PAY`.
- Під час відправлення — текст «ЗАЧЕКАЙТЕ…».

## 8. Модель даних

```ts
type DeliveryType = 'warehouse' | 'courier';

interface CheckoutInput {
  fullName: string;
  phone: string;
  email: string;
  quantity: number;

  city: string;
  cityRef: string;
  deliveryType: DeliveryType;

  // режим 'warehouse'
  warehouse: string;

  // режим 'courier'
  street: string;
  building: string;
  flat: string;
}
```

`lib/checkoutSchema.ts` експортує `checkoutSchema` (Zod) і виведений тип.
`CheckoutInput` у `lib/types.ts` приводиться до цієї форми.

## 11. Відкриті питання

Немає. Усі рішення зафіксовано в брейнштормі:

- Оплата — лише WayForPay; екран подяки + віджет відкриваються разом.
- Верхньої межі кількості немає.
- Популярні міста — константа без кешу.
- Точка доставки — відділення/поштомат (один список, відділення першими)
  або кур'єр (вулиця/будинок/квартира).
- Telegram-нотифікація і передача полів доставки — поза скоупом.
